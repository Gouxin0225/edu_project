# Edu Platform 系统功能测试报告

测试时间：2026-06-02 09:41-10:00（Asia/Shanghai）  
测试对象：`/edu-platform` 项目，后端 Spring Boot + MyBatis Plus，前端 Vue 3 + Vite  
测试方式：服务状态检查、自动化测试、前端生产构建、数据库抽样核对、角色登录、核心接口黑盒测试、权限边界测试、代码逻辑走查。

## 1. 测试结论

系统基础服务可启动，前端代理、后端接口、数据库、Redis 均可用；管理员、讲师、学生、班主任四类角色均可登录。核心只读功能链路整体可用，包括用户/班级管理、教师统计、题库、考试、问卷、模板、学生看板、学生班级、学生考试记录、错题本、班主任入口等。

本次发现的主要问题集中在统计口径和状态归类：

1. 管理员用户列表/统计把逻辑删除账号计入结果，导致接口统计和有效账号数不一致。
2. 学生看板把已提交但未批改的任务放入 `pendingTasks`，状态为 `SUBMITTED`，前端语义容易混淆。
3. 管理端统计中考试数量与数据库有效考试数存在口径差异，需要统一统计条件。
4. 前端生产构建成功，但多个 chunk 超过 500KB，后续应做代码分包优化。

## 7. 发现的问题

### 7.1 用户列表和统计包含逻辑删除账号

现象：

- 数据库有效用户：24。
- 数据库全部用户：30。
- `GET /admin/user/stats` 返回 `total: 30`，`teachers: 13`，`assistants: 4`。
- 数据库中 `is_deleted=1` 的账号有 6 个。

代码原因：

- `AdminUserController.listUsers()` 查询没有加 `is_deleted = 0`。
- `AdminUserController.getUserStats()` 使用 `selectCount(null)` 和仅按角色统计。
- `SysUser.isDeleted` 虽然导入了 `@TableLogic`，但字段上没有标注。

影响：

- 管理员看到的用户总数、分页总数和有效账号数不一致。
- 已删除讲师/班主任可能影响统计和运营判断。

建议：

- 给 `SysUser.isDeleted` 添加 `@TableLogic`，或在所有用户查询中明确追加 `.eq(SysUser::getIsDeleted, (byte) 0)`。
- 用户统计、用户列表、角色筛选保持同一过滤口径。
- 增加管理员用户列表和统计的单元/集成测试。

### 7.2 学生看板待办状态语义不清

现象：

- `GET /api/student/dashboard` 中，考试 `E2E_NORMAL_2026060101` 返回在 `pendingTasks` 内，但状态是 `SUBMITTED`。

代码原因：

- `StudentServiceImpl.getDashboard()` 对非 `GRADED` 的已有提交统一设置 `SUBMITTED`，并加入 `pendingTasks`。

影响：

- 学生可能把“已提交待批改”误解为“仍需处理”。
- 前端如果只按列表名称展示，会产生状态冲突。

建议：

- 将任务分组调整为 `upcomingTasks`、`todoTasks`、`submittedTasks`、`completedTasks`。
- 或保留 `pendingTasks`，但前端明确显示“待批改”，不要与“待完成”混用。

### 7.3 统计口径需要统一

现象：

- 数据库有效考试为 28。
- 管理员班级统计接口 overview 返回 `examCount: 29`。

建议：

- 明确统计是否包含逻辑删除、草稿、已发布、过期任务。
- 管理端、教师端、学生端统一任务状态枚举和统计口径。
- 对统计 SQL 增加固定样本测试，避免后续功能迭代引入口径偏差。

### 7.4 前端构建包体偏大

现象：

- `npm run build` 成功，但 `element-plus`、`echarts`、`vendor` 等 chunk 超过 500KB。

建议：

- 在 `vite.config.ts` 中增加 `manualChunks`。
- 对 ECharts、XLSX、Element Plus 等大依赖按路由懒加载。
- 后续加入构建产物大小阈值检查。

### 7.5 测试脚本执行方式存在环境限制

现象：

- 在当前沙箱中，直接执行 `curl` 可访问 `3001/8080`。
- 通过子 shell 或本地脚本执行相同 curl 命令时出现 `curl: (7) Couldn't connect to server`。

判断：

- 该问题更像测试执行环境限制，不是应用服务本身问题。
- 本报告的接口结论均来自直接 curl 命令结果。

修正：

- 已新增 `/edu-platform/scripts/system-smoke-test.sh` 作为统一系统冒烟测试入口。
- 脚本固定使用 `curl --noproxy '*'`，并统一校验后端、前端、前端代理和登录链路。
- 已将 Vite 开发代理目标固定为 `http://127.0.0.1:8080`，减少 `localhost` 在不同执行环境中的解析差异。
- 在 Codex/沙箱等限制子进程网络权限的环境中，脚本需在宿主 shell 或具备网络权限的执行环境运行；普通部署环境可直接执行。

## 8. 功能逻辑合理性分析

合理点：

- 四角色菜单和后端 `@RequireRole` 大体一致，越权访问能被拒绝。
- 登录后替换活跃 token，有利于控制同账号多端登录风险。
- 初始密码强制修改有后端拦截逻辑，允许 `/user/password` 和 `/auth/logout` 通过，流程合理。
- 考试、作业、问卷、错题本、统计形成了完整教学闭环。
- 学生班级申请、班主任审核、教师/管理员管理的职责拆分清晰。

需要调整的点：

- 用户软删除逻辑没有在所有查询中统一生效。
- 学生任务状态维度不够细，提交后待批改与待完成混在一起。
- 班级、考试、问卷统计口径需要产品化定义，否则不同页面会出现“数字都对但互相不一致”的问题。
- AI 功能接口可用性受外部 API key 影响，本次只验证了会话/知识库列表等本地接口，未验证真实模型调用。

## 9. 建议新增或优化的功能

1. 增加统一的“任务中心”
   - 学生侧按待开始、待提交、待批改、已完成、已过期分类。
   - 教师侧按草稿、已发布、待批改、已结束分类。

2. 增加统计口径说明和筛选器
   - 允许按班级、讲师、任务状态、时间范围筛选。
   - 页面展示“是否包含草稿/过期/已删除”的口径说明。

3. 增加软删除恢复与停用能力
   - 删除账号建议区分“停用登录”和“彻底删除/隐藏”。
   - 管理员可查看已停用账号并恢复。

4. 增加操作审计日志
   - 覆盖登录、创建考试、发布考试、批改、重置密码、删除用户等关键操作。
   - 便于追踪误操作和安全事件。

5. 增加批改工作台
   - 汇总所有待批改考试/作业。
   - 支持按班级、任务、学生筛选，并显示逾期/高风险标记。

6. 增加学生学习建议
   - 基于错题本和成绩统计生成知识点薄弱项。
   - AI 问答可以关联错题、知识库和复习计划。

7. 增加端到端自动化测试
   - 覆盖登录、创建考试、发布、学生提交、教师批改、成绩查询。
   - 建议使用 Playwright 或 Cypress，并准备独立测试数据库。

## 10. 后续测试建议

本次以只读和低风险验证为主，没有执行批量导入、删除、发布、提交、批改等强写入链路。建议下一轮在独立测试库中覆盖：

- 管理员创建/删除/重置用户。
- 教师创建题目、导入题目、创建考试、发布考试。
- 学生开始考试、保存答案、提交考试、防切屏记录。
- 教师批改主观题、发布成绩、导出成绩。
- 作业提交、打回、再次提交、批注、批改。
- 问卷创建、发布、学生提交、统计查看。
- 文件上传下载的类型、大小、安全边界。
- AI 真实调用、超时、取消、无 key 降级提示。

## 11. 总体评估

当前系统核心功能基本可用，权限边界有效，前后端构建和基础接口稳定。优先修复用户软删除统计问题和学生任务状态归类问题；随后统一统计口径，并补齐端到端自动化测试。这样能显著降低管理端数据不一致和学生端任务理解偏差。
